## 1. الرؤية والمشكلة

### المشكلة
محلات الخياطة الكبيرة في مالي وغرب أفريقيا تُدار بالدفاتر الورقية:
- طلبات تضيع أو تتأخر بلا تنبيه، ومقاسات الزبائن تُعاد كتابتها كل مرة.
- أجور الخيّاطين (بالقطعة) تُحسب يدوياً كل أسبوع — أخطاء ونزاعات.
- المالك لا يعرف ربحه الحقيقي (إيرادات − أجور − رواتب − مصاريف − تكلفة البضاعة).
- السكرتيرة تطّلع على أرقام مالية لا يريد المالك كشفها.

### الحل
تطبيق واحد (هاتف + متصفح) يدير الزبائن والمقاسات والطلبات والمواعيد والمخزون
والموظفين والمالية، مع **عزل مالي صارم** بين الدورين، وسرعة عالية على شبكات
ضعيفة، ويُباع لكل محل كنسخة معزولة تحمل اسمه وشعاره.

### مبادئ غير قابلة للتفاوض
1. **مستخدمان فقط**: الجيرون (MANAGER) بصلاحية كاملة، والسكرتيرة (SECRETARY)
   لا ترى أي بيانات مالية إطلاقاً (أرباح، إيرادات، مصاريف، أجور، رواتب).
2. **العزل المالي مفروض في الخادم** (فحص الدور في كل endpoint + قيود في قاعدة
   البيانات) وليس بإخفاء واجهات فقط؛ أي endpoint مالي يعيد 403 للسكرتيرة،
   ومجموعة اختبارات `backend/tests/` تثبت ذلك دائماً.
3. **السرعة أولوية #1**: ترقيم صفحات في كل قائمة، ضغط صور + مصغّرات،
   تحميل كسول، تخزين محلي، فهارس SQL.
4. **لا تسجيل ذاتي**: الحسابان يُنشآن عند تجهيز المحل ويديرهما الجيرون.
5. **كل جدول مالي append-only**: لا تعديل ولا حذف (محجوب بـ DB triggers)؛
   التصحيح صفّ جديد في جدول `*_corrections` بسبب إلزامي، والقيم الفعلية
   تُقرأ من views (`*_effective`).

## 2. المستخدمون والأدوار

| | الجيرون (Gérant) | السكرتيرة (Secrétaire) |
|---|---|---|
| الزبائن والمقاسات | ✅ كامل | ✅ كامل |
| الطلبات + الفواتير + واتساب | ✅ | ✅ (السعر الذي يعرفه الزبون فقط) |
| المواعيد والتقويم | ✅ | ✅ |
| المنتجات (عرض/بيع) | ✅ | ✅ تسجيل بيع (دون رؤية سجل المبيعات) |
| الخيّاطون: أسماء وتواصل | ✅ | ✅ |
| الخيّاطون: أجور وقِطع | ✅ | ❌ 403 |
| الموظفون الشهريون (رواتب) | ✅ | ❌ الوحدة غائبة كلياً |
| المالية (كل شيء) | ✅ | ❌ الوحدة غائبة كلياً |
| الإعدادات (اسم/شعار/كلمات مرور) | ✅ | ❌ |

## 3. الوحدات الوظيفية (شاشة رئيسية من 9 بلاطات + Staff)

### 3.1 الزبائن (Clients)
- CRUD كامل، بحث فوري مع debounce بالاسم أو الهاتف.
- مقاسات مرنة لكل نوع لباس (key-value/JSON وليس أعمدة ثابتة).
- سجل طلبات لكل زبون (المجموع يُقرأ من القيم الفعلية المشتقة).

### 3.2 المنتجات (Produits)
- فئات: عطور / أحذية / أقمشة؛ سعر بيع، سعر شراء (تكلفة)، مخزون، صور.
- البيع ذرّي ومُسعَّر من الخادم: السعر يُقرأ من القاعدة، الإجمالي يُحسب
  خادمياً، والمخزون يُخصم في نفس المعاملة (بشرط `quantity >= qty`).
  العميل لا يرسل أسعاراً أبداً.
- السكرتيرة تسجّل بيعاً لكن قائمة المبيعات للجيرون فقط (نمط write-only).
- تصحيح البيع (تعديل كمية أو إلغاء — جيرون فقط) يعيد ضبط المخزون بالفرق
  في نفس المعاملة.

### 3.3 الخيّاطون (Tailleurs) — أجر بالقطعة
- قائمة خيّاطين؛ نقرة تفتح ورقة واحدة فيها كل شيء: أسبوع حقيقي قابل
  للتنقّل (اثنين→أحد) بكل الأيام السبعة (الألبسة/الكميات/الزبائن) +
  إجمالي يومي وأسبوعي + إضافة إدخال + تصحيح + تعديل التعرفة.
- الإدخال اليومي مفصّل: نوع اللباس + عدد القطع + سعر القطعة (يُكتب مع كل
  إدخال) + ربط اختياري بطلب (اسم الزبون يُشتق من الطلب ولا يُعاد كتابته).
- عدة إدخالات لنفس اليوم مسموحة؛ التاريخ append-only (التصحيح يغيّر العدد
  فقط عبر سجل تصحيحات).
- الدفع أسبوعي؛ الأسابيع بمعيار ISO-8601 في الطرفين.

### 3.4 الموظفون الشهريون (Staff) — جيرون فقط
- وحدة مستقلة براتب شهري ثابت + سجل دفعات رواتب (append-only).
- السكرتيرة تُعاد توجيهها لو حاولت الدخول.

### 3.5 المالية (Finances) — جيرون فقط، غائبة كلياً عن واجهة السكرتيرة
- KPI: إيراد الطلبات المسلَّمة، مبيعات المنتجات، أجور الخيّاطين، الرواتب،
  المصاريف اليدوية (سبب+مبلغ+تاريخ)، تكلفة البضاعة المباعة (COGS)،
  **صافي الربح = الإيرادات − (أجور + رواتب + مصاريف + COGS)**.
- فلاتر: يوم / أسبوع / شهر / سنة / فترة مخصصة.
- جدول تفصيلي قابل للتوسيع لكل فئة يسرد عملياتها مع مجموع فرعي.
- تعديل المصاريف عبر `expense_corrections` فقط (سبب إلزامي).

### 3.6 الجاهز (Prêt-à-porter)
- موديلات جاهزة: اسم، قماش، صور، فيديو اختياري، سعر، تكلفة شراء.

### 3.7 الطلبات (Commandes)
- الطلب متعدد البنود (`order_items`، append-only): لكل بند كمية وسعر وحدة؛
  الإجمالي مشتق من view (البنود الملغاة = 0). تعديل البند عبر
  `order_item_corrections` بسبب إلزامي — لا تعديل في المكان.
- ربط بخيّاط، حالات: `en_attente → en_cours → terminé → livré`.
- عدة طلبات لنفس الزبون في نفس اليوم مسموحة.
- التسليم **ينقل** الطلب إلى الأرشيف (تغيير حالة، لا نسخ).
- **فاتورة PDF** بشعار المحل (A4، بنود، إجمالي/عربون/باقٍ، رابط الترويج)
  ومشاركة عبر الطابعة/الملفات + **إرسال واتساب** بملخص جاهز
  (`wa.me` مع بادئة مالي 223) — متاحان للدورين.

### 3.8 المواعيد (Rendez-vous)
- تقويم مدموج للقراءة فقط: مواعيد يدوية سابقة ∪ تاريخ تسليم كل طلب نشط
  (إنشاء الطلب يضع تسليمه في التقويم تلقائياً — الطلب هو مصدر الحقيقة).
- الأقرب أولاً؛ تحذير أحمر + عدّاد تنازلي لكل ما يبقى عليه ≤ 3 أيام؛
  النقر يفتح تفاصيل الطلب.

### 3.9 الأرشيف (Historique)
- الطلبات المسلَّمة بكامل تفاصيلها، بحث وفلترة بالزبون أو التاريخ.

### 3.10 الإعدادات (Paramètres) — جيرون فقط
- اسم المحل + الشعار (يظهران في شاشة الدخول والفواتير، قابلان للتغيير
  حيّاً؛ بديل بصري "R" متدرج إن لم يُرفع شعار).
- كلمات مرور الحسابين، التعرفة الافتراضية للقطعة، رابط مجموعة الترويج،
  لون الواجهة لكل محل (theme_color).

## 4. المتطلبات غير الوظيفية

| المجال | المتطلب |
|---|---|
| **الأداء** | ترقيم صفحات في كل القوائم؛ ضغط الصور وتوليد مصغّرات تلقائياً في `/api/upload`؛ فهارس على أعمدة البحث والتواريخ؛ يعمل مقبولاً على 3G |
| **الأمان** | JWT + bcrypt؛ الدور يُقرأ من جدول `users` في كل طلب (لا يُوثق من التوكن)؛ 403 خادمي لكل مالي؛ rate limiting على الدخول؛ HTTPS إلزامي في الإنتاج |
| **سلامة البيانات** | append-only بمحفّزات DB لكل الجداول المالية + جداول تصحيح + views فعلية؛ نسخ احتياطي يومي (قاعدة + صور) مع تدوير 14 يوماً وتنبيه بضرورة نسخة خارج السيرفر |
| **الاختبارات** | ‎100+ اختبار تكامل (Jest + Supertest على PostgreSQL حقيقية عبر embedded-postgres)؛ تشمل إثبات كل قواعد العزل المالي وappend-only وحسابات المالية |
| **التوافق** | Android (APK موقّع)، والويب للمتصفح (iPhone/حاسوب) من نفس الكود |

## 5. البنية التقنية (قرارات نهائية — لا تُعاد مناقشتها)

- **الواجهة**: Flutter + Provider + go_router، بنية feature-first
  (`lib/features/<feature>/{data,domain,presentation}`)، هوية لونية موحّدة
  (Deep Teal + Gold) عبر `AppColors`.
- **الخادم**: Node.js + Express + PostgreSQL في `backend/`؛ REST API؛
  الهجرات تعمل تلقائياً عند الإقلاع.
- **الاستضافة**: DigitalOcean VPS + Docker Compose + nginx (موجّه
  subdomains) + certbot (HTTPS).
- **لا Firebase** (قرار مغلق منذ 2026-07-05).

## 6. نموذج البيع — نسخة معزولة لكل محل

- **لا multi-tenancy ولا عمود `shop_id`**: كل محل له نشر Docker مستقل
  وقاعدة PostgreSQL خاصة على منفذ `API_PORT` فريد (3001، 3002…).
- التخصيص ديناميكي بالكامل (اسم، شعار، لون، تعرفة، رابط ترويج) — صفر
  تعديل كود لكل عميل؛ الـ APK يُبنى باسم ومعرّف `appId` خاصين بالمحل
  (لا يتغيّر بعد التسليم أبداً).
- **تجهيز محل جديد بأمر واحد**: `scripts/new-shop.sh` على السيرفر
  (منفذ + أسرار فريدة + clone + Docker + nginx + HTTPS + cron نسخ احتياطي
  + سطر CSV للسجل) ثم `scripts/build-shop-app.ps1` على جهاز التطوير
  (ويب + APK للمحل بأمر واحد).
- **تشغيل الأسطول**: `update-all.sh` يحدّث كل المحلات دفعة واحدة
  (نسخة أمان → سحب → إعادة بناء → فحص صحة)؛ سجل مركزي
  `docs/shops-registry-template.csv` يتتبع كل محل.
- الإيراد: بيع/اشتراك لكل محل (يُتفاوض عليه لكل عميل — عمود
  `monthly_fee_fcfa` في السجل).

## 7. خارج النطاق (الإصدار الحالي)

- تطبيق أو حساب للزبون النهائي (الزبائن سجلات فقط).
- multi-tenancy مشترك، دفع إلكتروني، إشعارات push، iOS APK
  (الويب يغطي iPhone)، تعدد لغات الواجهة (فرنسية فقط)، تعدد فروع
  للمحل الواحد.

## 8. مقاييس النجاح

- صفر تسريب مالي للسكرتيرة (تثبته الاختبارات في كل تغيير).
- إدخال طلب جديد في أقل من دقيقة؛ فتح أي قائمة في أقل من ثانيتين على 3G.
- إغلاق أسبوع الخيّاطين بلا حساب يدوي ولا نزاع.
- تجهيز محل جديد كامل في أقل من 30 دقيقة (سكربتات التجهيز).
- استعادة نسخة احتياطية مُختبرة لكل محل قبل اعتباره "في الإنتاج".
